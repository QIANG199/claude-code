## 一、状态管理（store.ts）— 35 行的极简设计

### 1.1 完整实现

整个状态管理只有 35 行代码，核心就是三个方法：

```mermaid

    class Store~T~ {
        +getState() T
        +setState(updater: T → T) void
        +subscribe(listener) unsubscribe
    }
    class createStore {
        <<factory>>
        +createStore(initialState, onChange?) Store
    }
    createStore --> Store : 创建
```

**类比 Java**：这就是一个极简版的 `EventBus` + `StateHolder`。

### 1.2 设计精髓

```mermaid
flowchart TB
    Input["setState(updater)"] --> Prev["prev = state"]
    Prev --> Next["next = updater(prev)"]
    Next --> Check{"Object.is(next, prev)?"}
    Check -- "相同，跳过" --> Skip(["无操作"])
    Check -- "不同" --> Update["state = next"]
    Update --> OnChange["onChange({newState, oldState})\n触发副作用"]
    Update --> Notify["for listener of listeners\nlistener()\n通知订阅者"]

    style Check fill:#E67E22,color:#fff
    style Update fill:#27AE60,color:#fff
```

三个设计要点：

| 要点 | 做法 | 为什么 |
|------|------|--------|
| 不可变更新 | `updater(prev) => next` | 避免意外修改旧状态 |
| 引用相等检查 | `Object.is(next, prev)` | 避免无意义的重渲染 |
| 副作用分离 | `onChange` 回调 | 状态变更后的副作用（日志、持久化）独立处理 |

**为什么遇到不安全工具后要全部串行？**

因为不安全工具（如 Edit）会改变文件状态，后续的"安全"工具（如 Read）可能需要读取修改后的内容。如果并行执行，Read 可能读到旧内容。

**类比 Java**：就像数据库事务中的读写锁——一旦有写操作介入，后续所有操作都必须排队等待，避免脏读。

## 三、中断机制 — Ctrl+C / ESC 是怎么工作的

### 3.1 整体架构

中断的核心是 JavaScript 的 `AbortController`，类似 Java 的 `Thread.interrupt()`，但更灵活。

```mermaid
sequenceDiagram
    participant U as 用户
    participant REPL as REPL.tsx
    participant AC as AbortController
    participant Q as query()
    participant API as Claude API (streaming)
    participant BT as BashTool (子进程)

    Note over U: 按下 Ctrl+C 或 ESC

    U->>REPL: 键盘事件
    REPL->>AC: abortController.abort('interrupt')

    Note over AC: signal.aborted = true<br/>signal.reason = 'interrupt'

    par 并行传播
        AC-->>API: signal 触发 → SDK 抛出 APIUserAbortError
        AC-->>BT: signal 触发 → 子进程特殊处理（不杀死）
        AC-->>Q: 下次检查 signal.aborted 时发现中断
    end

    API-->>Q: catch APIUserAbortError
    Q->>Q: yield "Interrupted by user"
    Q-->>REPL: return { reason: 'aborted_streaming' }
```

### 3.2 AbortController 基础

```typescript
// 创建
const abortController = new AbortController()

// 触发中断（传入原因字符串）
abortController.abort('interrupt')

// 检查状态
abortController.signal.aborted          // true
abortController.signal.reason           // 'interrupt'
```

**传递给 API 调用**：signal 被传给 Anthropic SDK，SDK 内部监听这个信号。一旦中断，正在接收的 HTTP 流会被立刻关闭。

**创建位置**（`src/utils/abortController.ts`）：

```typescript
export function createAbortController(
  maxListeners: number = 50,
): AbortController {
  const controller = new AbortController()
  setMaxListeners(maxListeners, controller.signal)
  return controller
}
```

还提供了 `createChildAbortController(parent)` 用于创建子 Agent 的中断控制器（见 3.7 节）。

### 3.6 场景二：BashTool 执行中断（命令正在运行）

#### 核心设计：不杀死子进程

```typescript
// src/utils/ShellCommand.ts:186
#abortHandler(): void {
    // On 'interrupt' (user submitted a new message), don't kill — let the
    // caller background the process so the model can see partial output.
    if (this.#abortSignal.reason === 'interrupt') {
        return  // ← 什么都不做！不杀死进程！
    }
    this.kill()  // 只有非 interrupt 的 abort 才杀死
}
```

#### BashTool 中断后的处理

```typescript
// src/tools/BashTool/BashTool.tsx:684

    && abortController.signal.reason === 'interrupt'
```

子进程被**后台化**（backgrounded），用户可以继续对话。下次对话时模型可以看到该命令的输出。

### 3.7 子 Agent 的中断传播

```mermaid

    subgraph forward["父 → 子：自动传播"]

        F2["propagateAbort() 触发"]

        F1 --> F2 --> F3
    end

    subgraph backward["子 → 父：隔离"]
        direction TB
        B1["子 abort()"]
        B2["父不受影响"]
    end

    Parent --> forward
    Child --> backward

    style Parent fill:#4A90D9,color:#fff
    style Child fill:#8E44AD,color:#fff
```

**实现**（`src/utils/abortController.ts`）：

```typescript
export function createChildAbortController(parent: AbortController): AbortController {
    const child = createAbortController()

    // 父已中断 → 子也立刻中断
    if (parent.signal.aborted) {
        child.abort(parent.signal.reason)
        return child
    }

    // WeakRef 防止内存泄漏
    const weakChild = new WeakRef(child)
    const weakParent = new WeakRef(parent)
    const handler = propagateAbort.bind(weakParent, weakChild)

    // 父中断时 → 子也中断
    parent.signal.addEventListener('abort', handler, { once: true })

    // 子中断时 → 清理父的监听器
    child.signal.addEventListener('abort',
        removeAbortHandler.bind(weakParent, new WeakRef(handler)),
        { once: true }
    )

    return child
}
```

**WeakRef 设计**：用弱引用避免父 Agent 持有已废弃的子 Agent 导致内存泄漏。如果子 Agent 被丢弃（没有 abort），它仍然可以被 GC 回收。

**类比 Java**：`WeakRef` 类似 Java 的 `WeakReference`——不阻止 GC 回收。传播机制类似 `CompletableFuture` 的 `cancel(true)` 向子任务传播。

### 3.8 submit-interrupt vs escape-interrupt

Claude Code 区分两种中断触发方式：

| 类型 | 触发方式 | signal.reason | 是否插入中断消息 |
|------|---------|---------------|-----------------|
| **escape-interrupt** | 按 ESC | `'interrupt'` | 否（用户没有新消息） |
| **submit-interrupt** | 输入新消息并回车 | `'interrupt'` | 否（新消息本身就够了） |
| **其他 abort** | 超时、错误等 | 其他值 | 是（插入 "Interrupted by user"） |

```typescript
// src/query.ts:1047
// 当 reason === 'interrupt' 时，不生成 "Request interrupted by user" 消息
// 因为用户的新消息会紧跟其后，或者用户只是按了 ESC 不需要额外消息
if (toolUseContext.abortController.signal.reason !== 'interrupt') {
    yield createUserInterruptionMessage({ toolUse: false })
}
```

### 3.9 流式空闲超时

除了用户主动中断，还有一个**自动超时机制**防止连接挂死：

```typescript
// src/services/api/claude.ts:1869
// 如果 API 流式响应 90 秒没有新数据到达，自动 abort
const STREAM_IDLE_TIMEOUT_MS =
    parseInt(process.env.CLAUDE_STREAM_IDLE_TIMEOUT_MS || '', 10) || 90_000

function resetStreamIdleTimer(): void {
    clearStreamIdleTimers()
    streamIdleTimer = setTimeout(() => {
        // 90 秒没收到数据 → 超时中断
        streamIdleAborted = true
        abortController.abort('stream_idle_timeout')
    }, STREAM_IDLE_TIMEOUT_MS)
}
```

每次收到新的 SSE chunk 时重置计时器。如果 90 秒内没有任何数据到达，自动 abort 防止会话永远挂起。

### 3.10 中断机制与 Java 的对比

| 机制 | Claude Code (JS) | Java |
|------|------------------|------|
| 中断信号 | `AbortController.abort(reason)` | `Thread.interrupt()` |
| 信号传播 | 手动通过 WeakRef 监听 | 自动传播（interrupt 状态） |
| HTTP 流中断 | SDK 内部关闭连接 | `InputStream.close()` |
| 子进程处理 | 不杀死，后台化 | 取决于实现 |
| 超时中断 | setTimeout + abort | `Future.cancel(true)` |
| 原因传递 | `signal.reason` 字符串 | 需要自定义字段 |

## 四、MCP 协议 — 外部工具接入

### 4.1 MCP 是什么

MCP（Model Context Protocol）是一个标准协议，让外部程序能向 Claude 暴露工具、资源和提示词。

**类比 Java**：MCP 就像 SPI（Service Provider Interface）——定义一套标准接口，第三方实现者按接口规范提供服务，框架自动发现和加载。

```mermaid
        S3["/verify: 验证代码"]
    end

    skill -- "告诉 Claude 按流程做事" --> Claude["Claude 按指令调用 tools"]
```

**类比 Java**：
- Tool 类似 Service 层的原子方法——做一件具体的事
- Skill 类似 Spring Batch Job 定义——它不是代码，而是一组编排好的指令，告诉 Claude "按这个流程做事"

## 六、QueryEngine — 非交互式模式

### 6.1 什么是 QueryEngine

QueryEngine 是 `query()` 的高级封装，用于**非交互式**场景（SDK 调用、自动化脚本）。

**类比 Java**：
- REPL 模式 类似 Spring MVC 的 `@Controller`（交互式，有 UI）
- QueryEngine 模式 类似 Spring 的 `CommandLineRunner`（非交互式，无 UI）

### 6.2 QueryEngine vs REPL

| 维度 | REPL | QueryEngine |
|------|------|-------------|
| 交互方式 | 终端 UI（Ink/React） | SDK 调用 |
| 用户输入 | 键盘输入 | 编程传入 |
| 输出方式 | 终端渲染 | 回调/流 |
| 状态管理 | React state | 简单变量 |
| 权限确认 | 弹窗询问 | 预设/自动 |

### 6.3 QueryEngine 核心流程

```mermaid
flowchart TB
    Input["QueryEngine.query(userMessage)"] --> CTX["构建 ToolUseContext\n(工具列表、权限、MCP)"]
    CTX --> SP["构建 system prompt\ngetSystemPrompt() + userContext + systemContext"]
    SP --> MSG["构建消息列表\n[历史消息..., 新用户消息]"]
    MSG --> Query["调用 query()\nAsyncGenerator 流式输出"]

    Query --> Y1["yield StreamEvent — 流式文本"]
    Query --> Y2["yield Message — 完成的消息"]

```

## 八、设计模式总结

| 模式 | 在哪里用 | Java 等价 |
|------|---------|-----------|
| Command | Tool 接口 | `Command.execute()` |
| Strategy | 工具编排的并行/串行策略 | `Strategy` 接口 |
| Observer | store.ts 状态订阅 | `EventListener` |
| Adapter | MCPTool 包装 MCP 工具 | `Adapter` 模式 |
| Template Method | Skill.getPromptForCommand | `AbstractMethod` |
| Composite | system prompt 拼装 | `Builder` 模式 |
| Singleton | context.ts 的 memoize | `@Scope("singleton")` |
| Chain of Responsibility | 中断信号的父子传播 | `Filter Chain` |
| Background Task | BashTool 子进程后台化 | `Daemon Thread` |

## 九、一句话总结

- **store.ts**：35 行极简状态管理，不可变更新 + 引用相等检查 + 副作用分离
- **工具编排**：只读工具并行（最大 10），有副作用工具串行，遇到第一个不安全工具后全部强制串行
- **中断机制**：AbortController 一把梭——API 流立刻关闭，BashTool 子进程不杀死而是后台化，父子 Agent 通过 WeakRef 单向传播
- **MCP**：标准协议接入外部工具，适配器模式包装成内部 Tool，对 Claude 透明
- **Skill**：预编排的指令集，不是代码而是 Prompt + 工具权限的组合，声明式定义工作流
- **QueryEngine**：query() 的非交互式封装，REPL / SDK / Pipe 三种模式共用同一个核心引擎
