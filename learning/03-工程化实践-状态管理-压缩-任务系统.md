## 一、状态管理（store.ts）— 35 行的极简设计

### 1.1 完整实现

整个状态管理只有 35 行代码，核心就是三个方法：

```mermaid
classDiagram
    class Store~T~ {
        +getState() T
        +setState(updater: T → T) void
        +subscribe(listener) unsubscribe
    }
    class createStore {
        <<factory>>
        +createStore(initialState, onChange?) Store
    }
    createStore --> Store : 创建
```

**类比 Java**：这就是一个极简版的 `EventBus` + `StateHolder`。

### 1.2 设计精髓

```mermaid
flowchart TB
    Input["setState(updater)"] --> Prev["prev = state"]
    Prev --> Next["next = updater(prev)"]
    Next --> Check{"Object.is(next, prev)?"}
    Check -- "相同，跳过" --> Skip(["无操作"])
    Check -- "不同" --> Update["state = next"]
    Update --> OnChange["onChange({newState, oldState})\n触发副作用"]
    Update --> Notify["for listener of listeners\nlistener()\n通知订阅者"]

    style Check fill:#E67E22,color:#fff
    style Update fill:#27AE60,color:#fff
```

三个设计要点：

| 要点 | 做法 | 为什么 |
|------|------|--------|
| 不可变更新 | `updater(prev) => next` | 避免意外修改旧状态 |
| 引用相等检查 | `Object.is(next, prev)` | 避免无意义的重渲染 |
| 副作用分离 | `onChange` 回调 | 状态变更后的副作用（日志、持久化）独立处理 |

## 二、上下文压缩 — 对话太长怎么办

### 2.1 问题背景

Claude API 有上下文窗口限制（如 200k tokens）。当对话历史超过限制时，必须压缩。

**类比 Java**：类似数据库分页——你不能把所有历史记录都放在一个查询里。

### 2.2 压缩策略层次

```mermaid
flowchart TB
    Raw["原始消息历史"]

    subgraph L1["Layer 1: applyToolResultBudget()"]
        direction LR
        L1d["裁剪单个工具结果"]
        L1c["代价: 零 (只是截断字符串)"]
    end

    subgraph L2["Layer 2: snipCompact()"]
        direction LR
        L2d["截断过早的历史"]
        L2c["代价: 低 (丢失早期上下文)"]
    end

    subgraph L3["Layer 3: microcompact()"]
        direction LR
        L3d["已缓存的工具结果可以删除"]
        L3c["代价: 低 (利用 prompt cache)"]
    end

    subgraph L4["Layer 4: autocompact()"]
        direction LR
        L4d["用小模型 (Haiku) 总结历史"]
        L4c["代价: 中 (额外 API 调用)"]
    end

    Raw --> L1 --> L2 --> L3 --> L4 --> Compact["压缩后的消息"]

    style Raw fill:#7F8C8D,color:#fff
    style L1 fill:#27AE60,color:#fff

    style L3 fill:#F39C12,color:#fff

```

### 2.3 autocompact 详细流程


## 三、工具编排（toolOrchestration.ts）— 并行与串行

### 3.1 核心问题

Claude 一次回复可能包含多个 tool_use 块。怎么执行它们？

```mermaid
flowchart LR
    subgraph reply["Claude 回复包含 4 个 tool_use"]
        direction TB
        T1["Read(file1.ts) — 只读"]
        T2["Read(file2.ts) — 只读"]
        T3["Edit(file1.ts) — 有副作用"]
        T4["Bash(npm test) — 有副作用"]
    end

    style reply fill:#34495E,color:#fff
```

### 3.2 执行策略

```mermaid
flowchart TB
    Input["runTools(toolUseBlocks)"] --> Partition["partitionToolCalls() — 分区"]

    Partition --> Safe["并行组\nisConcurrencySafe = true\nRead, Grep, Glob, WebFetch"]
    Partition --> Unsafe["串行组\nisConcurrencySafe = false\nEdit, Write, Bash"]

    Safe --> Concurrent["runToolsConcurrently()\n同时执行多个只读工具\n最大并发数: 10"]
    Unsafe --> Serial["runToolsSerially()\n一个接一个执行"]

    Concurrent --> Yield1["yield 结果 (按完成顺序)"]
    Serial --> Yield2["yield 结果 (按执行顺序)"]

    Yield1 --> Final["合并结果"]
    Yield2 --> Final

    style Partition fill:#E67E22,color:#fff
    style Safe fill:#27AE60,color:#fff
    style Unsafe fill:#E74C3C,color:#fff
    style Concurrent fill:#2ECC71,color:#fff
    style Serial fill:#C0392B,color:#fff
```

### 3.3 分区算法

```mermaid
flowchart TB
    Start["遍历所有 tool_use 块"] --> Check{"isConcurrencySafe(input)?"}

    Check -- "true" --> Parallel["放入并行组"]
    Check -- "false" --> Serial["放入串行组"]
    Serial --> Force["之后的所有工具都强制串行\n(即使它们本身是安全的)"]

    Parallel --> Next["下一个"]
    Force --> Next
    Next --> Start

    style Check fill:#E67E22,color:#fff
    style Parallel fill:#27AE60,color:#fff
    style Serial fill:#E74C3C,color:#fff
```

**为什么遇到不安全工具后要全部串行？**

因为不安全工具（如 Edit）会改变文件状态，后续的"安全"工具（如 Read）可能需要读取修改后的内容。如果并行执行，Read 可能读到旧内容。

## 四、QueryEngine — 非交互模式

### 4.1 什么是 QueryEngine

QueryEngine 是 `query()` 的高级封装，用于**非交互式**场景（SDK 调用、自动化脚本）。

**类比 Java**：
- REPL 模式 ≈ Spring MVC 的 `@Controller`（交互式，有 UI）
- QueryEngine 模式 ≈ Spring 的 `CommandLineRunner`（非交互式，无 UI）

### 4.2 QueryEngine vs REPL

| 维度 | REPL | QueryEngine |
|------|------|-------------|
| 交互方式 | 终端 UI（Ink/React） | SDK 调用 |
| 用户输入 | 键盘输入 | 编程传入 |
| 输出方式 | 终端渲染 | 回调/流 |
| 状态管理 | React state | 简单变量 |
| 权限确认 | 弹窗询问 | 预设/自动 |

### 4.3 QueryEngine 核心流程

```mermaid
flowchart TB
    Input["QueryEngine.query(userMessage)"] --> CTX["构建 ToolUseContext\n(工具列表、权限、MCP)"]
    CTX --> SP["构建 system prompt\ngetSystemPrompt() + userContext + systemContext"]
    SP --> MSG["构建消息列表\n[历史消息..., 新用户消息]"]
    MSG --> Query["调用 query()\nAsyncGenerator 流式输出"]

    Query --> Y1["yield StreamEvent — 流式文本"]
    Query --> Y2["yield Message — 完成的消息"]
    Query --> Y3["yield ToolUseSummary — 工具摘要"]

    Y1 --> Collect["收集结果"]
    Y2 --> Collect
    Y3 --> Collect

    Collect --> Save["recordTranscript() — 保存记录"]
    Save --> Flush["flushSessionStorage() — 持久化"]
    Flush --> Return["返回最终结果"]

    style Input fill:#4A90D9,color:#fff
    style Query fill:#8E44AD,color:#fff
    style Return fill:#27AE60,color:#fff
```

## 七、如果我要做一个简化版 Agent

基于这三天的学习，一个最小可运行的 agent 需要：

### 必须有（核心骨架）

| 模块 | 你需要做什么 | 参考文件 |
|------|------------|---------|
| Tool 接口 | 定义 `name`, `inputSchema`, `call()` | `Tool.ts` |
| 2-3 个 Tool 实现 | Bash, FileRead, FileEdit | `BashTool.tsx`, `FileReadTool.ts` |
| query 循环 | while(true) { 调API → 执行tool → 拼消息 } | `query.ts` |
| API 客户端 | 调 Anthropic SDK 流式接口 | `claude.ts` |
| system prompt | 拼装身份 + 工具描述 + 上下文 | `prompts.ts` |

### 最好有（工程化增强）

| 模块 | 你需要做什么 | 参考文件 |
|------|------------|---------|
| 权限系统 | 哪些工具需要用户确认 | `types/permissions.ts` |
| 上下文压缩 | 对话太长时自动压缩 | `services/compact/` |
| 工具并行 | 只读工具并行执行 | `toolOrchestration.ts` |
| 状态管理 | 全局状态 + 订阅机制 | `store.ts` |

### 可以后做（高级功能）

| 模块 | 你需要做什么 | 参考文件 |
|------|------------|---------|
| MCP 支持 | 接入外部工具 | `services/mcp/client.ts` |
| Skill 系统 | 预编排指令集 | `skills/` |
| 子 Agent | 在 tool 内启动另一个 agent | `AgentTool/` |
| 记忆系统 | 跨会话持久化 | `memdir/` |
